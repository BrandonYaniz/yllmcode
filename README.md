# yllmcode

yllmcode is a local-first software planning and implementation harness. It is
designed to help turn a rough project idea into a structured plan, record the
important decisions, and then execute implementation work in small, reviewable
phases.

The project is intentionally CLI-first. Future interfaces can build on the same
core service layer, but the command line should remain a stable interface for
planning, review, and implementation workflows.

## Status

This repository is in early bootstrap.

The first implemented command is `yllmcode init`, which creates a project-local
`.yllmcode` directory for planning documents, decision records, phase files,
policies, prompts, tests, and related project memory.

Current scope:

- Go command-line application
- local project initialization
- conservative filesystem behavior
- no daemon
- no model-provider implementation
- no internet access
- no automatic dependency installation

## What yllmcode Creates

Running `yllmcode init` in a project will create a `.yllmcode` directory shaped
for planning and phased implementation:

```text
.yllmcode/
  project.md
  instructions.md
  architecture.md
  roadmap.md
  dependencies.md
  .gitignore
  decisions/
  maps/
  phases/
  policies/
  prompts/
  research/
  tests/
```

Initialization should preserve existing files by default. Re-running the command
should report skipped files instead of overwriting project notes.

## Repository Layout

The planned Go layout keeps CLI parsing separate from the service logic:

```text
cmd/
  yllmcode/
    main.go
internal/
  service/
  project/
go.mod
```

Future packages will be added as the planning, phase, patch, test, and yllmd
integration layers become real implementation work.

## Install

The project is not published as a packaged release yet. During development,
build from the repository:

```sh
go build ./cmd/yllmcode
```

Run the initializer from a project directory:

```sh
go run ./cmd/yllmcode init
```

## Development

Run the full test suite before sending changes for review:

```sh
go test ./...
```

Keep changes small and phase-oriented. The project favors boring, auditable Go
over clever abstractions.

## Safety Model

yllmcode should protect the user's repository by default:

- do not overwrite project files unless explicitly asked
- do not modify files outside the selected project root
- do not install dependencies automatically
- do not run commands generated by model output without review
- do not add network behavior in bootstrap phases
- do not commit automatically from product code

## Versioning

Date-based versions use `YYYY.MM.DD`.

Release versions append `-Release`.

If multiple non-release builds are needed on the same day, use a numeric suffix:

```text
2026.07.02.01
2026.07.02.02
```
